CREUS — 제품 소개

기획서 한 장으로 시작해, AI가 생성한 영상 구조 위에서 팀과 협업하며 결과물을 제작하는 AI 영상 제작 협업 워크스페이스

핵심 가치 제안

  • 기획서(텍스트/PDF) → AI가 Scene-Cut 구조와 제작 가이드 자동 생성
  • 컷 단위로 가이드라인·프롬프트·결과물 관리
  • 팀원 실시간 협업 (누가 어디를 작업 중인지 추적)
  • "영상 1편 = 수십 컷의 분산 작업물"을 한 캔버스에서

타겟 사용자

영상 콘텐츠 제작팀
(크리에이터, 프로덕션 하우스)

출시 형태

외부 퍼블릭 (누구나 가입)
+ B2B 영업

MVP 완성 · 내부 도그푸딩 완료
→ 출시 준비 진입

4월 · 토대 다지기

테마 — 크리에이티브팀이 쓸 수 있는 수준으로 안정화하면서, 출시까지의 전체 실행 계획을 확정한다.

개발 작업

피그마 비충돌 항목 우선

  • P0 긴급 버그 3건 수정 (전역규칙 삭제, 프로젝트 설명 저장, 아이콘 겹침)
  • P1 버그/개선 (AI 분석 한글 번역, Restore 미반영, 버전 라벨, Scene 일괄 삭제)
  • 디자인 시스템 적용 (진행 중)

계획 작업 (이달 핵심 산출물)

  • 피그마 이미지 기반 기능 명세서 완성
  • 5~8월 상세 작업 계획 + 전체 일정 산정
  • 새 디자인 방향 검토
4월 말 체크포인트   긴급 버그 0건 · 기능 명세서 완성 · 실행 계획서 확정

5월 · 기술 기반 + 협업 인프라

테마 — 외부 사용자를 받기 위한 기술 인프라를 깐다. 이 위에 6~7월의 기능이 올라간다.

기술 기반

  • AI 기획서 확정 플로우 트랜잭션 보강
  • 테스트 인프라 구축 시작 (크리티컬 경로부터)
  • DB/인프라 정비

협업 인프라

  • 알림 시스템 기초 (Socket.IO 위에 — 리뷰 요청, 멤버 관련)
  • 전역 규칙 자동 상속 (Page → Cut)

UI + 신기능

  • 디자인 시스템 적용 마무리
  • 새 디자인 방향 확정·적용
  • 피그마 기반 신기능 착수 (우선순위 항목)
5월 말 체크포인트   알림이 동작하는 협업 시연 가능 · 기술 기반 안정화

6월 · 핵심 가치 + 시연 시작

테마 — AI 생성 기능과 핵심 신기능을 구현하고, 대외 시연을 시작한다.

핵심 작업

  • 시연용 안정 브랜치 분리 (6월 초)
  • AI 이미지 생성 v1 — 최소 1개 모델 연동으로 "기획서 → AI 분석 → 이미지 생성"까지 도구 안에서 완결
  • 권한/역할 분리 — 최소 Owner/Editor (B2B 조직 단위 검토)
  • 피그마 신규 모듈 구현 (명세 우선순위)
  • 활동 추적 코드 심기 (데이터 수집 시작)

이 달의 의미

제품의 가장 큰 격차(AI 생성 미연동)를 해소하는 달.

이 시점부터 CREUS의 핵심 가치 제안이 실제로 동작합니다.

출시

8월 말 ~ 9월 중

CREUS — 리스크와 대응

1. 피그마 신기능 물량

대표님 피그마 추가 기능 상당량, 6~7월에도 추가 가능성

대응: capacity 70%만 계획, 30%는 버퍼. 모든 기능을 "출시 필수" vs "출시 후"로 분류 후 경영진 합의.

2. AI 생성 모델 선정·연동

어떤 모델을 어떤 방식으로 연동할지 미정 (API, 비용, 에러 핸들링)

대응: 5월 중 모델 선정 + API 검증 완료. 1개 모델(이미지)로 시작, 검증 후 확장.

3. 인력 (2~3명)

전체 scope 대비 인력 빠듯, 병렬 처리 한계

대응: 월별 scope를 2~3명 기준으로 현실 유지. 시연 전 필수 / 출시 전 필수 / 출시 후 가능으로 역산 배치.

4. 시연과 개발 병행 (6월~)

6월부터 시연 시작, 불안정 버전 노출 위험

대응: 시연용 안정 브랜치 분리 운영. 개발은 메인에서, 시연은 안정 브랜치에서.

CREUS — 의사결정 요청 사항

확정 필요  출시 일정에 직접 영향

1. 피그마 기능 분류 합의

"출시 필수(Must)" vs "출시 후 업데이트(Nice/Post)" 합의 → 출시 시점 확정

2. AI 생성 모델 선정

이미지 생성 모델 선택 (비용, 품질, 라이선스)

3. 프라이싱/결제 시점

출시 시점에 결제까지 vs 초기 무료/제한 운영

방향 확인  전략 방향 설정

4. B2B 대응 범위

팀/조직 관리, 프로젝트 공유 어디까지 출시에 포함

5. 인증 확장

Google OAuth 유지 vs 이메일 가입 추가

6. 챗봇(AI 어시스턴트) 범위

제품 내 AI 작업 도우미의 범위와 시점

7. Replit 인프라 한계

동시 사용자, 스토리지 비용 등 상용화 검토

Promptly — 유지보수 계획 (부록)

기본 전제 — CREUS가 주력 개발 트랙. Promptly는 메인 1명 최소 운영 체제로 서비스 안정성과 TIPS 과제 의무를 지킨다.

Phase 1

버그 수정

구독/플랜/결제/크레딧 알려진 버그 해결

매출 직결 결제·구독 안정성

Phase 2

코드 건강성

storage.ts God Object 분리, 컨트롤러 분할, 타입 정리, 핵심 경로 테스트

운영 사고 예방 + 유지보수 속도

Phase 3

운영 개선

Admin UI, 오래된 AI 모델 교체, 모델 교체 자동화, 보안 강화

최소 인원 운영 가능 체제

Phase 4

TIPS 과제

정부 과제 의무 이행 (27.10 완료). Phase 1~3과 병행.

정부 과제 의무 이행

관통 항목   긴급 운영 이슈(장애, 보안, 외부 API 변경)는 전 Phase에 걸쳐 상시 대응
